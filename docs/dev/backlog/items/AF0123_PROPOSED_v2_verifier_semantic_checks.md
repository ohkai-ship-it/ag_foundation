## Problem

V1Verifier performs only mechanical checks: required/optional step pass/fail, error field presence, and schema validation evidence. It cannot evaluate whether an output is *semantically correct* — whether it actually answers the user's question, is complete, or is consistent with its sources.

Example: a summarization skill can return valid JSON with a `summary` field, pass all V1 checks, yet produce a summary that misses key points or contradicts the source material. V1 has no way to detect this.

---

## Goal

- V2Verifier adds LLM-powered semantic verification on top of V1 mechanical checks
- Three semantic checks: **relevance**, **completeness**, **consistency**
- LLM verification is **additive** — V1 runs first, V2 layers on top
- If LLM is unavailable or times out, V1 result stands (graceful degradation)
- Semantic evidence is recorded in trace alongside mechanical evidence

---

## Non-goals

- Replacing V1Verifier (V2 extends, does not replace)
- Auto-correcting outputs (that's V2Executor's job)
- Policy enforcement (deferred — Gate C condition #1)
- Acceptance criteria generated by V3Planner (V3 is a parallel AF; V2Verifier works without it)

## Design

Based on the V2 roadmap in **AF-0115** (V1Verifier, DONE).

### Architecture

```
V2Verifier.verify(trace)
  ├── V1Verifier.verify(trace)          # mechanical first (fast, deterministic)
  │   └── per-step: required/optional, error field, schema
  └── LLM semantic evaluation           # additive layer (slower, intelligent)
      ├── relevance_check(step, task)    # does output answer the task?
      ├── completeness_check(step, task) # are expected sections present?
      └── consistency_check(step, sources) # does output contradict sources?
```

### Semantic check definitions

| Check | Input | LLM Prompt Goal | Output |
|-------|-------|-----------------|--------|
| Relevance | step output + original task | "Does this output address the task?" | score 0.0–1.0 + reason |
| Completeness | step output + task + skill metadata | "Are expected elements present?" | score 0.0–1.0 + missing elements |
| Consistency | step output + source inputs (if available) | "Does the output contradict its sources?" | score 0.0–1.0 + contradictions |

### Trace evidence model

```python
class SemanticVerification(BaseModel):
    relevance_score: float        # 0.0–1.0
    relevance_reason: str
    completeness_score: float     # 0.0–1.0
    completeness_missing: list[str]
    consistency_score: float      # 0.0–1.0
    consistency_issues: list[str]
    overall_pass: bool            # True if all scores above threshold
    llm_model: str
    llm_tokens_used: int
    evaluation_ms: int

# Added to existing verification evidence
class VerificationEvidence(BaseModel):
    # ... existing V1 fields ...
    semantic: SemanticVerification | None  # None if LLM unavailable
```

### Thresholds (configurable per workspace)

- Relevance: ≥ 0.6 (default)
- Completeness: ≥ 0.5 (default)
- Consistency: ≥ 0.7 (default)
- Overall: all three must pass

### Graceful degradation

- LLM call timeout (default 10s): V1 result used, `semantic: null` in evidence
- LLM provider error: same — V1 result stands
- Token budget exceeded: skip semantic checks for remaining steps

## Risks

| Risk | Impact | Mitigation |
|------|--------|------------|
| LLM adds 1-3s latency per step | Slower runs | Bounded timeout (10s), skip on budget exceeded |
| LLM semantic scores are non-deterministic | Flaky verification | Threshold-based (not exact match), configurable per workspace |
| Extra LLM cost per run | Token budget | Semantic checks only on steps that pass V1 (don't LLM-check already-failed steps) |
| Test isolation with LLM calls | CI contamination | Provider stubs in tests (existing pattern) |

## LLM integration (CRITICAL — follow existing patterns)

V2Verifier needs an `LLMProvider` injected via constructor (same pattern as V2Planner):

```python
from ag.providers.base import LLMProvider, ChatMessage, MessageRole

class V2Verifier(V1Verifier):
    """LLM-powered semantic verification layered on V1 mechanical checks."""

    def __init__(self, provider: LLMProvider | None = None) -> None:
        super().__init__()
        self._provider = provider  # None = graceful degradation to V1-only

    def verify(self, trace: RunTrace) -> tuple[str, str | None]:
        # Step 1: V1 mechanical checks (always runs)
        status, message = super().verify(trace)

        # Step 2: LLM semantic checks (additive, skippable)
        if self._provider is None:
            return status, message
        semantic = self._run_semantic_checks(trace)
        # Merge semantic results into evidence
        return self._merge_results(status, message, semantic)

    def _run_semantic_checks(self, trace: RunTrace) -> SemanticVerification:
        """Run relevance/completeness/consistency checks via LLM."""
        messages = [
            ChatMessage(role=MessageRole.SYSTEM, content=self._semantic_system_prompt()),
            ChatMessage(role=MessageRole.USER, content=self._semantic_user_prompt(trace)),
        ]
        response = self._provider.chat(messages=messages)
        return self._parse_semantic_response(response.content)
```

**Wiring in runtime/CLI** — V2Verifier receives the same provider instance:
```python
# In src/ag/cli/main.py or create_runtime()
provider = get_provider(ProviderConfig(provider="openai", model="gpt-4o-mini"))
verifier = V2Verifier(provider)         # ← replaces V1Verifier()
orchestrator = V1Orchestrator(executor, verifier, recorder)
```

**Graceful degradation:** `V2Verifier(provider=None)` behaves identically to `V1Verifier()` — tests that don't need LLM instantiate without provider.

### Default pipeline version update (CRITICAL)

- `src/ag/core/runtime.py` `create_runtime()`: currently hardcodes `V1Verifier()` — **change to `V2Verifier(provider)`**
