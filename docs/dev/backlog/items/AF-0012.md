# AF-0012 — CLI_REFERENCE surface parity v0.1: missing run options + missing subcommand stubs (+ config tests)
# Version number: v0.2

## Metadata
- **ID:** AF-0012
- **Type:** Feature
- **Status:** Ready
- **Priority:** P2
- **Area:** CLI
- **Owner:** Jacob
- **Target sprint:** Sprint 02+

## Problem
The CLI implements core commands, but several options and subcommands documented in CLI_REFERENCE are missing (BUG-0002 run options, BUG-0003 subcommands). This creates spec drift and blocks API-parity planning. Additionally, config.py has low coverage (~46%); adding CLI surface for config should include config tests.

## Goal
Establish CLI surface parity for v0.1 by adding missing `ag run` options and stubbing missing subcommands with clear behavior, while increasing confidence via focused CLI + config tests.

## Non-goals
- Implementing full behavior for every stubbed subcommand (OK to stub).
- Enabling real web access (`--allow-web`) (can be stubbed as always-false).
- Adding LLM provider integration (separate items).

## Acceptance criteria (Definition of Done)
- [ ] `ag run` supports: `--file <path>`, `--task <task.json>`, and `--confirm/--no-confirm` per CLI_REFERENCE (help documents precedence).
- [ ] `--file` reads file content into prompt; respects workspace path safety rules; fails clearly if file missing.
- [ ] `--task` accepts a TaskSpec JSON file and validates via TaskSpec schema (invalid rejected clearly).
- [ ] `--confirm` is dev-safe: prompts user before run; in `--json` mode, fail with structured error unless `--no-confirm`.
- [ ] Add stubs for missing subcommands (minimum): `ag runs tail`, `ag ws config get/set`, `ag artifacts open/export`, `ag skills test/enable/disable`, `ag playbooks validate/set-default`.
- [ ] Stub semantics are consistent: non-zero exit code + message `Not implemented in v0`; with `--json`, return structured JSON error.
- [ ] Tests added/updated (CLI surface):
- [ ]   - `ag run --file` happy path + missing-file error case.
- [ ]   - `ag run --task` happy path + invalid-task error case.
- [ ]   - `--confirm` TTY behavior test (non-interactive should fail unless `--no-confirm`), including `--json` structured error.
- [ ]   - Stub command presence tests: help shows subcommands; running them returns expected not-implemented behavior (human + json).
- [ ] Config tests added (config.py coverage focus):
- [ ]   - Default config path resolution (and override via env var if supported).
- [ ]   - Missing config file behavior (returns defaults or clear error—document which).
- [ ]   - Invalid YAML/JSON config produces clear error (structured JSON error if `--json`).
- [ ] After merge, config.py coverage increases measurably (target: +30pp or more for that file; exact % depends on implementation).
- [ ] BUG-0002 and BUG-0003 can be closed or reclassified with evidence.

## Implementation notes
- Keep parsing/behavior in CLI adapter; core runtime still receives TaskSpec and runs.
- For `--task`, use `TaskSpec.from_json()`; precedence: CLI `--workspace` overrides file unless explicitly forbidden.
- Centralize stub helper (e.g., `not_implemented(cmd_name, json_mode)`), so all stubs behave the same.
- For config tests: create temp config files in tests; avoid relying on user home directory; prefer env var override to point at temp.

## Risks
Medium: more surface area and precedence confusion. Mitigate with explicit help text and tests for precedence and error cases.

## PR plan (PR-sized)
1. PR (feat/cli-run-options): Add `--file`, `--task`, `--confirm/--no-confirm` + tests.
2. PR (chore/cli-stub-surface): Add missing subcommand stubs + JSON error semantics + tests.
3. PR (test/config-coverage): Add focused config tests (path resolution, missing/invalid config) and wire into CI.
